## 二、面试官高频问答

### Q1: 介绍一下这个项目的技术架构？

**回答框架**:

整个项目是 Next.js 16 App Router 全栈架构。前端核心是一个CSS 3D卡牌旋转环，78张塔罗牌均匀分布在一个虚拟圆环上，通过调整 `rotateY` 角度实现环的旋转。

手势识别这块我选型对比过两个方案：最初接了百度智能云手势识别API，但发现云端方案有200-500ms网络延迟，QPS限制导致频繁429，体验很差。后来改用MediaPipe Hands WASM——它在浏览器本地运行，21个手部关键点检测30fps零延迟，而且完全免费。我通过PIP关节角度(>155°判定伸直)实现手指状态分类：五指全伸直=旋转手势，仅食指伸直=选择手势。

AI调用方面，我用Vercel AI SDK v6的`streamText`接入DeepSeek V4 Pro。解牌和追问是两个独立路由，追问路由加了Function Calling——`getCardDetail`工具让AI按需查卡牌数据，而不是每次请求携带全部78张牌的信息，这样节省了大量token。上下文管理做了从后向前裁剪，中文按~4 chars/token估算，保持在7000 tokens以内。

部署在Vercel，绑了自定义域名`soe.9vvn.com`，CNAME解析到Vercel DNS，HTTPS自动签发。

### Q2: 你怎么设计Prompt的？有什么心得？

**回答框架**:

Prompt设计我认为最关键的一点是：**不要把Prompt当成一段静态文本，而是当成一个需要动态组装的函数**。我的解牌Prompt分为5层：

1. **人格层**(静态) — "星辰之镜"的人设，温柔但不故弄玄虚，明确禁止预测未来
2. **牌阵层**(动态) — 根据用户选的牌阵(3张/10张)注入每个位置的含义
3. **牌义层**(动态) — 把用户实际抽到的牌(正逆位、关键词、象征意义)注入，而不是78张全放
4. **输出格式** — 5段式结构(开场→逐牌→综融→指引→结语)，保证体验一致
5. **伦理边界** — 危机引导、不替代专业建议

设计思路来自我对AI行为的观察：给AI太多自由度，输出就会漂移。给它明确的结构约束，输出就更可控。另外我刻意要求**纯文本、禁止Markdown和表情符号**，因为我在实践中发现DeepSeek经常会输出`**粗体**`和大量emoji，人眼看很出戏——所以在代码层又加了一层`stripMarkdown()`兜底。

追问对话的Prompt策略不同：我把**完整解读上下文作为System Prompt重新注入**，这样追问时AI能引用之前说过的话，保持人格连贯性。这比简单的多轮对话效果好很多。

### Q3: 流式响应你是怎么实现的？遇到了什么问题？

**回答框架**:

服务端用Vercel AI SDK v6的`streamText()`，它返回的`toTextStreamResponse()`是`Content-Type: text/plain`的流。客户端我没有用`useCompletion` hook，而是**手动实现ReadableStream解析**：

```typescript
const reader = response.body.getReader();
const decoder = new TextDecoder();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  setCompletion(prev => prev + decoder.decode(value, { stream: true }));
}
```

选择手动实现的原因是：AI SDK v6的React hooks接口变动频繁，而且手动实现能让我更好控制AbortController，做中断、重试等错误处理。

遇到的主要问题是AI SDK v6的breaking changes：v6把`maxTokens`改成`maxOutputTokens`，`toDataStreamResponse`改成`toTextStreamResponse`，`useChat`的API也从`input/handleInputChange`模式变成了`sendMessage`模式。这些都是我在vibe coding过程中逐步排查解决的——这也是AI全栈开发的一个重要能力：**能快速定位AI SDK版本差异导致的兼容性问题**。

### Q4: 手势识别为什么不继续用百度API了？

**回答框架**:

实测后发现三个致命问题：
1. **延迟** — 云端API往返200-500ms，用户手势响应很迟钝
2. **QPS限制** — 百度免费额度QPS=2，我300ms截帧时每秒3.3次请求，大量429错误
3. **手势种类不匹配** — 百度识别的是静态手势分类(祈祷、OK、拳头)，但我的应用需要的是"手指伸直/弯曲"这种连续状态判断

换到MediaPipe后：30fps零延迟、不受QPS限制、完全免费。我在浏览器端通过PIP关节角度实现了手指状态实时分类，比云端方案体验好太多。

这个决策体现了我在AI应用开发中的一个原则：**能跑在本地的AI能力就别上云**。

### Q5: Function Calling你是怎么用的？

**回答框架**:

在追问对话中启用了`getCardDetail`工具。场景是：用户在解牌后追问"权杖八逆位具体是什么意思？"，AI可以调用工具查到详细牌义再回答，而不是我每次请求都把78张牌的完整数据放进prompt。

实现上用的是AI SDK v6的`tool()`函数，配合Zod定义input schema，`stopWhen: stepCountIs(3)`限制最多3轮工具调用防止死循环。

这个设计带来的好处：单次追问的input token从可能的5000+降到~3000，节省了40%的token成本。

### Q8: 这个项目最大的技术挑战是什么？

**回答框架**:

最大的挑战是**3D卡牌旋转环的手势交互闭环**。具体来说有几个难点：

1. **CSS 3D变换的数学计算**: 78张牌均匀分布，`frontIndex`的计算公式需要同时考虑环的旋转角度`angle`和每张牌自己的`i * step`角度，组合公式是`getFrontIndex = round((360 - angle) / step) % 78`。这个公式我最初写成了`round(angle / step)`，导致中心卡牌判断错误，后来通过加日志发现组合角度不对才修正。

2. **翻牌时机控制**: 用户在旋转中快速切换手势选牌时，卡牌的停止→吸附→翻转三个动作需要严格按顺序执行。我通过"pending selection"机制——先stop再等吸附动画完成再选——解决了中心卡空白的问题。

3. **opacity优先级**: 翻牌时中心卡应该明亮，周围卡虚化，已选卡半透明。这个优先级链是`isFlipping > anyFlipping > isSelected`，调试过程中因为顺序不对导致中心卡也被虚化。

### Q9: 你如何看待AI占卜类产品的伦理问题？

**回答框架**:

我在Prompt设计中有明确的伦理边界：
- 明确声明"塔罗是镜子而非预言书"，不声称能预测未来
- 遇到危机信号(自伤/自杀)引导寻求专业心理帮助
- 不提供医疗、法律、金融建议
- 承认塔罗只是参考，最终选择权在用户

技术层面，前端渲染前会`stripMarkdown()`去掉格式符号，让AI输出更像自然的对话而非营销文案。另外所有解读结果都是基于用户实际抽到的卡牌，结合正逆位含义给出，不是凭空生成的。

### Q10: 如果要把这个项目做成真正的移动App，你会怎么做？

**回答框架**:

有几个方向：
1. **PWA方案**(最快) — 加`manifest.json`和Service Worker，Vercel自动支持，可以添加到手机主屏幕，离线也能用基本功能
2. **React Native WebView** — 把现有Web代码嵌入WebView，打包成iOS/Android App
3. **原生Swift** — 重写iOS端，但MediaPipe有iOS SDK可以直接用，AI调用部分保持现有API路由

如果时间允许，我会先用PWA验证移动端体验，再考虑原生方案。产品迭代的节奏很重要——先验证后重写。

这个项目我实践了完整的部署链路：域名注册→DNS CNAME解析→Vercel部署→HTTPS自动签发。GitHub作为代码仓库，`.gitignore`排除敏感信息，环境变量通过Vercel管理。日志通过Vercel Logs和路由内`console.log`追踪。Token用量在API路由的`onFinish`回调中记录。

### "多模态/工作流编排"
这个项目本身就是多模态应用——手势(视觉)+AI文本(语言)。Function Calling机制本质上就是一个简单的工具编排：AI判断需要什么信息→调用工具→拿到结果→继续生成。如果要做更复杂的编排，我可以扩展到LangGraph，比如加入根据用户历史解读记录来个性化后续解读的流程。
